**Lesson Title:** Building Accountable AI Tools — One Project, Seven Moves

**Audience & Context:** Graduate students in INFO 7375 (Branding & AI), entering
the AI-build world from a marketing/design background — comfortable with brands,
new to pipelines, recipes, and validation. Format: a ~50-minute guest lecture /
online module that maps the whole semester arc before any single exercise.

## Learning Outcomes

By the end, learners will be able to:

1. **Explain** why fluency is the first sign of trouble — why a polished AI
   artifact is not yet a trustworthy one.
2. **Trace** one project across the arc (brand → target → data → brief → tool),
   naming what each exercise adds.
3. **Distinguish** a recipe that *runs* from one that is *conductor-ready* (knows
   when to stop and who decides).
4. **Locate** the human's irreducible work at each gate, rather than delegating
   the judgment with the task.
5. **Apply** the recipe lifecycle (DRAFT → VERIFIED) to decide what evidence
   promotes a claim.

## Slide Blueprint

### Slide 1 — Cold open: the fluent draft

- **Purpose:** Gain attention; plant the central tension before any definitions.
- **On-slide:** "It looked done. / It ran. / Nobody had checked."
- **Visual Type:** Interpretive (metaphor made concrete).
- **Visual Prompt:** Hand-drawn doodle, black ink on white. A glossy, perfectly
  formatted report page with a green checkmark — and a single thin crack running
  down its middle. No background.
- **Speaker Notes:** Open in a scene: an AI hands you a clean draft — labels,
  phases, the rhythm of expertise. The surface says finished. This whole course
  is about the gap between *polished* and *trustworthy*.

### Slide 2 — The one idea: fluency ≠ trust

- **Purpose:** State the thesis (problem → concept).
- **On-slide:** "Fluency is the first sign of trouble. / Trust is earned, not formatted."
- **Visual Type:** Interpretive.
- **Visual Prompt:** Two side-by-side cards — left "Fluent" (smooth, glowing),
  right "Trustworthy" (stamped, initialed, with a small human figure signing).
  An arrow from left to right labeled "work that can't be delegated."
- **Speaker Notes:** The machine is fluent for free. Trust costs human judgment.
  Everything ahead is a method for the part you cannot hand off.

### Slide 3 — The arc is one project, not seven assignments

- **Purpose:** Pre-training — give the map before the parts (segmenting).
- **On-slide:** "Brand → Target → Data → Brief → Tool" · "each move extends the last"
- **Visual Type:** Transformational (pipeline / timeline).
- **Visual Prompt:** Horizontal 7-node pipeline: A1 Brand · 1A Audit · A2 PRD ·
  A3 Build · A5 Brief · 5A Wrap / 5B Recipe. A faded dashed node "A4" between A3
  and A5 marked "build session — coming." Each node a labeled box, single accent color.
- **Speaker Notes:** Don't read this as seven unrelated homeworks. It's one
  brand-reputation project carried end to end. Note the dashed A4 — a real gap we
  name honestly, not hide.

### Slide 4 — Exercise 1: your brand as data

- **Purpose:** Concept — the self becomes structured, attestable artifacts.
- **On-slide:** "resume.json · brand.yml · gaps.md" · "aspiration → audience → position"
- **Visual Type:** Organizational (concept map).
- **Visual Prompt:** A central node "brand/" branching to three labeled files:
  resume.json, brand.yml, gaps.md; brand.yml expands to aspiration → audience →
  Dunford-frame positioning. Clean tree, labels on nodes.
- **Speaker Notes:** Before tools, you make yourself legible: a positioning built
  from aspiration and evidence, and a gaps file where every gap maps to the
  Madison build that would close it. The semester project falls out of the top row.

### Slide 5 — Exercise 1A: every claim must trace

- **Purpose:** Concept — introduce the trace-and-adjudicate gate (the core habit).
- **On-slide:** "Board claim → brand/ artifact?" · "human adjudicates every row"
- **Visual Type:** Transformational (flow with a gate).
- **Visual Prompt:** Flow: Figma board → "board-claims.json" → trace table
  (claims vs artifacts) → a gate icon (a small human stamp) → "resolved." One row
  highlighted red: "untraceable."
- **Speaker Notes:** Madison reads your Figma board over the API and checks that
  every visual claim traces to a `brand/` artifact. The machine proposes the
  trace; you adjudicate each row. That human stamp is the gate — remember it; it
  returns everywhere.

### Slide 6 — Pause & Reflect

- **Purpose:** Germane load — consolidate before the build half.
- **On-slide (prompt):** "Think of an AI output you shipped recently. Name one
  claim in it that you never actually traced to a source. What would have caught it?"
- **Visual Type:** None (reflection slide; plenty of white space).
- **Visual Prompt:** Minimal — a single hand-drawn question mark over a small
  figure, lots of empty space for the prompt text.
- **Speaker Notes:** Give them 60–90 seconds, written or pair-share. We're moving
  from describing the self to building the machine.

### Slide 7 — Exercise 2: target → gap → PRD → recipe

- **Purpose:** Concept — turn a real demand into a defended spec.
- **On-slide:** "targets.json → gap table → PRD → recipe @ DRAFT" · "+ a WON'T-build list"
- **Visual Type:** Transformational (step diagram).
- **Visual Prompt:** Four-step left-to-right: a job posting → SOC/O*NET-tagged
  targets.json → gap table → a PRD doc → a recipe file stamped "DRAFT." A small
  side box "WON'T build" with two crossed-out items.
- **Speaker Notes:** You capture a real posting, the machine maps it, you select
  and attest the top targets. The PRD is *defended* from resume + aspiration +
  target — no fun-but-orphaned build. The architecture lands as a recipe at DRAFT.

### Slide 8 — The spine: the recipe lifecycle

- **Purpose:** Pre-training for the rest — the status ladder that governs everything.
- **On-slide:** "DRAFT → SPECIFIED → RUNNABLE-SAMPLE → RUNNABLE-LIVE → VERIFIED" ·
  "a status is a claim; evidence promotes it"
- **Visual Type:** Relational/Organizational (rising staircase).
- **Visual Prompt:** A hand-drawn ascending staircase, each step labeled with a
  lifecycle stage; a small evidence tag (📎) sits on each step. A figure can only
  climb a step by placing a tag.
- **Speaker Notes:** This is the backbone. A DRAFT is a hypothesis; only VERIFIED
  carries evidence the intent is achievable. You never edit the status field —
  you earn it with a logged artifact. Watch the exercises climb this staircase.

### Slide 9 — Exercise 3: gather, validate, defend

- **Purpose:** Concept — the two-layer data gate (the build).
- **On-slide:** "data/raw → [validate] → data/verified" · "dedup · dates · critical-field · coercion audit"
- **Visual Type:** Transformational (pipeline with a filter).
- **Visual Prompt:** Two labeled buckets — "raw" (messy, overflowing) and
  "verified" (clean) — connected by a funnel labeled with the four checks. Nothing
  passes the funnel without a check mark.
- **Speaker Notes:** Three-plus Tier-1 sources in; nothing reaches `verified`
  without passing the gate. This is where quantity-over-quality dies structurally.

### Slide 10 — The data defense

- **Purpose:** Concept — the forcing function (germane: argument, not just output).
- **On-slide:** "Conformance (machine) + Adequacy (human)" · "Why exactly N rows?"
- **Visual Type:** Relational (two-column comparison).
- **Visual Prompt:** Split panel: left "Machine — conformance numbers" (a small
  table of pass rates); right "Human — adequacy argument" (a speech bubble: "why
  N rows? fitness? what did cleaning assume?").
- **Speaker Notes:** You can't defend what you didn't validate. The grade is the
  argument: conformance is the machine's half, adequacy is yours — the fitness
  scorecard and the cleaning report with the *assumption* column filled.

### Slide 11 — Pause & Reflect

- **Purpose:** Germane load before the strategy half.
- **On-slide (prompt):** "Predict: when you hand an AI a vague instruction, where
  does the vagueness go? Write one sentence before the next slide."
- **Visual Type:** None.
- **Visual Prompt:** Minimal doodle — a small figure handing a blank page to a
  large question mark. White space for the answer.
- **Speaker Notes:** Hold their prediction; the next slides pay it off — vagueness
  is *hallucinated*, not flagged.

### Slide 12 — Exercise 5: the Conductor Brief

- **Purpose:** Concept — one intent, two audiences, two forms.
- **On-slide:** "brief.md (for the human) + config (for the machine)" · "the sentence is the seam"
- **Visual Type:** Organizational (two-branch split from one root).
- **Visual Prompt:** One node "The Sentence" splitting into two: left a narrative
  scroll labeled "boss/client — why," right a JSON/YAML block labeled "conductor —
  rules." Both read the same sentence at the seam.
- **Speaker Notes:** The boss wants narrative intent; the AI wants declarative
  rules. Same intent, two forms. The one-sentence positioning is the seam where
  both audiences read the same words. This literacy gap is the AI-age skill.

### Slide 13 — Prose is the alibi; config is the actuator

- **Purpose:** Deep-dive — the named bug class and the divergence log.
- **On-slide:** "Vagueness is hallucinated, not filled in" · "ALIGNED / DRIFT / GAP"
- **Visual Type:** Relational (reconciliation table).
- **Visual Prompt:** A three-row table: each row a narrative claim vs config
  reality, tagged ALIGNED (green), DRIFT (amber), GAP (red). One DRIFT row circled:
  "grades source reliability — lived only in the prose."
- **Speaker Notes:** Run the conductor against the config, not the prose. The
  graded artifact is the divergence log. The differentiator you're proudest of is
  worthless until it's in the actuator — "you don't have a differentiator, you
  have a paragraph."

### Slide 14 — The A5 fork: visible vs accountable

- **Purpose:** Concept — frame 5A and 5B as a real choice, not two flavors.
- **On-slide:** "5A: make the tool visible (a URL a stranger uses)" · "5B: make it
  accountable (a recipe that knows when to stop)"
- **Visual Type:** Relational (side-by-side comparison).
- **Visual Prompt:** Two panels sharing one root "Your tool": left a browser
  window with a stranger's cursor (5A); right a recipe file with labeled stop-gates
  (5B). A fork-in-the-road icon between them.
- **Speaker Notes:** Same tool, two ways to make it real. 5A exposes it to a real
  user; 5B makes it a recipe a conductor can run and audit. Pick your path; the
  underlying project is identical.

### Slide 15 — Exercise 5A: the real user is the conductor

- **Purpose:** Concept (Path A) — usability divergence as evidence.
- **On-slide:** "Deploy → 3 real strangers → usability divergence log" · "their confusion is the run"
- **Visual Type:** Transformational (test loop).
- **Visual Prompt:** A loop: tool URL → three stick figures testing → notes
  (promised vs actual behavior) → a sharpened "One Thing" + a 2×2 positioning grid.
- **Speaker Notes:** You wrap the tool in a fenced template and ship it. Three
  strangers use it. What the brief *promised* the user would do vs. what they
  *actually* did is the human version of the config-silence GAP. Fence discipline
  (a clean diff) is graded.

### Slide 16 — Exercise 5B: conductor-ready is named-stops

- **Purpose:** Concept (Path B) — the deep idea of the whole arc.
- **On-slide:** "4 layers · 5 judgment types (PA/PF/TO/IJ/EI) · typed TODOs · audit loop"
  · "'human reviews here' is an unfinished gate"
- **Visual Type:** Transformational + Organizational (layered pipeline with gates).
- **Visual Prompt:** Four stacked layers (external → data/raw → data/verified →
  logs/reports), a contract between each; at one gate a door with a name tag, at
  another a "wall with no door" circled red ("who decides? unnamed").
- **Speaker Notes:** A recipe is finished not when it runs clean but when every
  place it must stop names *which human decides what*. The machine finds the gaps
  and proposes typed TODOs; you name the judgment type and close them. "A wall with
  no door" is the failure to avoid.

### Slide 17 — The honest gap: A4, the build session

- **Purpose:** Intellectual honesty — name what isn't built yet.
- **On-slide:** "A4 = walk verified data from RUNNABLE-SAMPLE → VERIFIED" · "the next exercise"
- **Visual Type:** Transformational (the staircase, one step empty).
- **Visual Prompt:** The lifecycle staircase from Slide 8, with the
  RUNNABLE-SAMPLE → VERIFIED step drawn as a dashed, empty step labeled "A4 —
  coming," a figure pausing before it.
- **Speaker Notes:** The arc has a real hole: the build session that promotes the
  agent consuming your data. We mark it as an empty slot, not a hidden one —
  modeling exactly the honesty the course teaches.

### Slide 18 — Synthesis: the irreducible part

- **Purpose:** Enhance retention/transfer — return to the thesis, now earned.
- **On-slide:** "Machine: fluent work. / Human: the gate, the judgment, the stop." ·
  "Trust is earned, not formatted."
- **Visual Type:** Interpretive (callback to Slide 2).
- **Visual Prompt:** The cracked report from Slide 1 — now with a small human
  figure placing an evidence stamp over the crack, sealing it. Caption: "checked."
- **Speaker Notes:** Every exercise put the machine on the fluent work and you on
  the irreducible work: tracing the claim, validating the data, naming who decides,
  knowing when to stop. That's the method. Carry it past this course.
